#cyberark-labs 


![[safe-permissions.png]]

Think of a **Safe** in CyberArk like a **super-secure bank vault**.

- The **Safe** stores privileged accounts (Administrator, Oracle, Windows, Linux, etc.).
    
- People (or groups) become **members** of the Safe.
    
- **Permissions** decide **what each member is allowed to do**.
    

## Think of it like a real bank vault

Imagine a bank vault.

Different employees have different jobs.

|Person|What they can do|
|---|---|
|Customer|Open their own safety deposit box|
|Bank Teller|Help customers|
|Vault Manager|Add or remove boxes|
|Security Guard|Watch cameras and logs|
|Auditor|Read logs but cannot change anything|

CyberArk works exactly the same way.

Instead of giving **everyone Full Access**, you give **only the permissions they need**.

This follows the **Principle of Least Privilege**.

> Give people **only the minimum permissions** needed to do their job.

---

# Why do Safe permissions exist?

Without permissions...

Imagine everyone could:

- Delete accounts
    
- Change passwords
    
- View Administrator passwords
    
- Remove Safe members
    

That would be extremely dangerous.

Instead CyberArk asks:

> "What exactly should this user be allowed to do?"

That's what these permissions control.

---

# 1. Access Permissions

![[access-safe-permissions.png]]

These are the permissions people use **every day**.

They simply allow someone to **use accounts**.

---

## List Accounts

> Allows members to view the accounts inside the Safe.

Without this permission...

The Safe would look empty.

Example:

Safe:

```
Windows Administrator
Oracle DBA
Linux Root
SQL Service
```

If you have **List Accounts**

You can see

```
Windows Administrator
Oracle DBA
Linux Root
SQL Service
```

If not...

```
(No accounts visible)
```

---

### Why do we need it?

You cannot use an account if you cannot even see it.

---

## Use Accounts

This is one of the most important permissions.

It allows users to connect through

- PSM
    
- PSMP
    

Example

Without seeing the password...

```
CyberArk
     │
     ▼
PSM
     │
     ▼
Windows Server
```

The user clicks

```
Connect
```

CyberArk logs in for them.

The password is never shown.

---

### Why do we need it?

Many companies never allow administrators to know passwords.

Instead they only connect through PSM.

---

## Retrieve Accounts

This permission is different.

Instead of logging in through PSM...

CyberArk actually shows the password.

Example

```
Username:
Administrator

Password:
MySecretPassword123
```

The user can

- copy it
    
- view it
    
- paste it somewhere
    

---

### Why do we need it?

Sometimes software cannot use PSM.

Example

A database tool may require entering the password manually.

Then the DBA retrieves the password.

---

# Difference

**Use Account**

```
User
   │
   ▼
PSM
   │
   ▼
Server

Password hidden
```

---

**Retrieve Account**

```
CyberArk

Password:
**********
```

Password becomes visible.

---

# 2. Account Management Permissions

![[account-management-safe-permissions.png]]

These permissions are mainly used by

- CyberArk Administrators
    
- Safe Owners
    
- Account Managers
    

They change the accounts stored in the Safe.

---

## Add Accounts

Allows someone to store a new account.

Example

Before

```
Windows Admin
Linux Root
```

Add

```
Oracle DBA
```

Now

```
Windows Admin
Linux Root
Oracle DBA
```

---

### Why?

New servers are constantly being added.

Someone must register their accounts.

---

## Update Account Properties

Changes information **about** the account.

Examples

- Platform
    
- Address
    
- Username
    
- Port
    
- Description
    

Notice

It **does NOT change the password.**

Example

Old

```
Server:
Server01
```

New

```
Server:
Server02
```

---

## Update Account Content

This changes the password **inside the Vault only**.

Imagine

Server password

```
Summer2026!
```

Vault password

```
OldPassword
```

You manually update the Vault.

Now

Vault

```
Summer2026!
```

The actual server password is **not changed**.

---

### Why?

Useful when synchronizing passwords after an external change.

---

## Initiate CPM Account Management Operations

This lets a user tell the CPM:

> Change the password now.

or

> Verify it now.

or

> Reconcile it now.

Normally CPM runs automatically.

But sometimes you want it immediately.

---

Example

```
Click

Change Password Now
```

CPM starts immediately.

---

## Specify Next Account Content

Normally CPM creates a random password.

Example

```
$T4k!9QxL2#
```

Sometimes you want the next password to be something specific.

Example

```
Winter2026!
```

The next CPM rotation uses that password.

This is rarely used.

---

## Rename Accounts

Changes the account name inside CyberArk.

Example

Before

```
Oracle Admin
```

After

```
Oracle DBA Production
```

---

## Delete Accounts

Deletes the account from the Safe.

Be careful.

It removes only the Vault record.

It does **not** delete the operating system account.

---

## Unlock Accounts

Sometimes someone opens an account and never closes it.

CyberArk locks it.

An administrator can unlock it.

---

# 3. Safe Management and Monitoring

![[safe-management-monitoring-safe-permissions.png]]

These permissions affect the **Safe itself**, not the accounts inside it.

---

## Manage Safe

Allows changing Safe settings.

Example

Rename Safe

```
Old

Windows
```

to

```
Windows Production
```

Change retention settings.

Change descriptions.

---

## View Safe Members

Allows seeing

Who has access?

Example

```
John
Read Only

Sarah
Full Control

CPM
Manager
```

---

## Manage Safe Members

This is a very powerful permission.

It allows

- Add users
    
- Remove users
    
- Change permissions
    

Example

```
Add DBA Team

Permission

Use Accounts
```

Now the DBA team has access.

---

## View Audit Log

Shows everything that happened.

Example

```
John viewed password

10:00

Sarah connected

10:15

CPM changed password

11:00
```

Very useful during investigations.

---

## Back Up Safe

Allows creating a backup of the Safe.

Usually used by administrators.

---

# 4. Workflow Permissions

![[workflow-safe-permissions.png]]

Some companies require approval before someone can use an account.

Example

Engineer requests

```
Administrator
```

CyberArk says

```
Waiting for approval...
```

---

## Confirm Requests

Allows approving those requests.

Example

John requests

```
Windows Admin
```

Manager clicks

```
Approve
```

John can now use it.

---

### Level 1 Confirmer

First approver.

---

### Level 2 Confirmer

Second approver.

Some companies require

```
Engineer
      │
      ▼
Manager Approval
      │
      ▼
Security Approval
      │
      ▼
Access Granted
```

Level 1 = Manager

Level 2 = Security

---

## Access Safe Without Confirmation

Allows skipping the approval process.

Usually given to emergency administrators.

---

# 5. Advanced Permissions

![[advanced-safe-permissions.png]]

These control folders inside the Safe.

---

## Move Accounts/Folders

Move accounts between folders.

Example

Before

```
Production

Windows01
```

Move to

```
Testing

Windows01
```

---

## Create Folders

Allows organizing accounts.

Example

```
Windows

Linux

Oracle

SQL
```

Folders make large Safes easier to navigate.

---

## Delete Folders

Deletes folders (not necessarily the accounts inside them, depending on the operation).

# Putting It All Together

Imagine a company has three employees:

### 👨‍💻 IT Administrator

Permissions:

- ✅ List Accounts
    
- ✅ Use Accounts
    
- ✅ Retrieve Accounts
    
- ✅ Add Accounts
    
- ✅ Update Properties
    
- ✅ Trigger CPM
    
- ✅ Manage Safe
    
- ✅ View Audit Logs
    

They can manage the Safe and its accounts.

---

### 👨‍🔧 Database Administrator (DBA)

Permissions:

- ✅ List Accounts
    
- ✅ Use Accounts
    
- ❌ Add Accounts
    
- ❌ Delete Accounts
    
- ❌ Manage Safe
    

They can connect to the database but cannot modify the Safe.

---

### 👩‍💼 Auditor

Permissions:

- ✅ List Accounts
    
- ✅ View Audit Log
    
- ❌ Retrieve Password
    
- ❌ Use Accounts
    
- ❌ Delete Accounts
    

They can review activity but cannot access or change privileged accounts.

---

## An easy way to remember the categories

|Category|Think of it as...|Examples|
|---|---|---|
|**Access**|**Can I use the account?**|List, Use, Retrieve|
|**Account Management**|**Can I modify the account?**|Add, Delete, Rename, Change properties|
|**Safe Management & Monitoring**|**Can I manage the Safe itself?**|Manage Safe, Members, Audit Logs|
|**Workflow**|**Can I approve or bypass approval requests?**|Confirm Requests, Access Without Confirmation|
|**Advanced**|**Can I organize folders?**|Create, Move, Delete Folders|

As you continue your CyberArk lab, you'll notice that different built-in roles (such as **Auditor**, **CPM**, **PSM**, and **Vault Administrators**) receive different combinations of these permissions. CyberArk separates them so that each component and each person can perform only the tasks required for their role, which greatly improves security and reduces the risk of accidental or malicious changes.
