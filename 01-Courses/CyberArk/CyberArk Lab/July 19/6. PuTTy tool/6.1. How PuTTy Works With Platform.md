## First: What is a Platform in CyberArk?

A **Platform** in CyberArk is a set of rules and settings that tells CyberArk:

> "How should I manage this type of account?"

Example:

You have a Linux administrator account:

```
Username: root
Server: Linux01
Password: ********
```

CyberArk needs to know:

- How do I connect to this server?
    
- How do I change the password?
    
- How often should I rotate the password?
    
- What commands should I use?
    
- Which CPM plugin should I use?
    

The Platform contains these rules.

Example:

```
Platform:
Unix via SSH

Settings:
-------------------------
Connection method: SSH
Protocol: SSH
Port: 22
Password rotation: Every 30 days
CPM Plugin: UnixSSH
PSM: Enabled
```

---

# Where does PuTTY fit?

Without CyberArk:

```
Administrator
      |
      |
    PuTTY
      |
      | SSH
      |
      v
 Linux Server
```

The administrator opens PuTTY manually:

```
Host:
192.168.10.20

Port:
22

Username:
root
```

Then logs in.

---

# With CyberArk + Platform + PSM

The flow changes:

```
User
 |
 |
PVWA
 |
 |
CyberArk Platform
 |
 |
PSM
 |
 |
SSH connection
 |
 |
Linux Server
```

The user does **not** open PuTTY and type the password.

---

## Step-by-step example

Imagine you have this account stored in CyberArk:

```
Safe:
Linux-Admins

Account:
root@Linux01

Platform:
Unix via SSH
```

---

### Step 1 — User opens PVWA

User logs into CyberArk:

```
https://cyberark.company.com/PasswordVault
```

They see:

```
Account:
root@Linux01

Actions:
[Connect]
[Retrieve Password]
```

---

### Step 2 — User clicks "Connect"

CyberArk looks at the Platform:

```
Platform:
Unix via SSH
```

CyberArk says:

"Okay, this account uses SSH."

The Platform tells CyberArk:

```
Use SSH
Port 22
Use PSM
Hide password
Record session
```

---

### Step 3 — PSM starts the connection

PSM creates a connection:

```
PSM Server
     |
     |
     | SSH
     |
     v
Linux Server
```

PSM uses the stored account:

```
Username:
root

Password:
********
```

The user never sees the password.

---

### Step 4 — Where PuTTY comes in

There are two possibilities.

---

## Case 1: Normal PuTTY (outside CyberArk)

Administrator uses:

```
PuTTY
 |
 SSH
 |
 Linux Server
```

CyberArk is not involved.

---

## Case 2: PuTTY through CyberArk PSM

In some CyberArk environments, PuTTY is configured as a **connection component**.

The flow becomes:

```
User
 |
 |
PVWA
 |
 |
PSM
 |
 |
PuTTY connection component
 |
 |
SSH
 |
 |
Linux Server
```

Here PuTTY is running **inside the PSM server**, not on the user's computer.

The user sees:

```
CyberArk Remote Session Window
```

Inside it:

```
PuTTY terminal
-----------------

Linux01 login:

[root@Linux01 ~]#
```

CyberArk records everything:

- commands typed
    
- screen activity
    
- session duration
    
- user identity
    

---

# Simple analogy

Imagine a bank vault.

### Platform = instruction manual

It says:

> "For Linux accounts, use this key, this door, and this security process."

---

### PuTTY = the vehicle

It is just the car that drives you to the server.

---

### PSM = security guard

The guard:

- checks your permission
    
- opens the door
    
- watches what you do
    
- records everything
    

---

So the relationship is:

```
Platform
   |
   | tells CyberArk HOW to connect
   |
   v
PSM
   |
   | uses connection method (SSH/PuTTY)
   |
   v
Linux Server
```

### The key thing to remember for your CyberArk lab:

> **The Platform does not control PuTTY. The Platform tells CyberArk which connection method to use. PSM uses that information to create the session, and PuTTY may be the SSH client used behind the scenes.**
