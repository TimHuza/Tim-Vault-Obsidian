# CyberArk Dual Approvals Explained (Beginner Level)

Imagine you have a **very powerful key** that can open the door to a bank vault.

Normally, if you have the key, you can open the vault yourself.

But the bank says:

> "For extra security, one person cannot open the vault alone. Another trusted person must approve it first."

That idea is **Dual Approval** in CyberArk.

In simple words:

> **Dual Approval means one person requests access to a privileged account, and another person must approve that request before access is granted.**

It is a **security control to prevent unauthorized use of powerful accounts.**

---

# Why do we need Dual Approval?

CyberArk manages **privileged accounts**, such as:

- Domain Administrator
    
- Root account
    
- Database administrator
    
- Firewall administrator
    
- Server administrator
    

These accounts are extremely powerful.

Example:

A user named **John** has permission to use the Domain Admin account.

Without Dual Approval:

```
John
 |
 | Clicks "Connect"
 |
 v
CyberArk gives Domain Admin password
 |
 v
John logs into server
```

The problem:

- John could access the account anytime.
    
- Nobody checks why he needs it.
    
- A stolen user account could be abused.
    

---

With Dual Approval:

```
John
 |
 | Requests Domain Admin access
 |
 v
CyberArk creates approval request
 |
 v
Manager/Security team approves
 |
 v
CyberArk allows access
 |
 v
John connects
```

Now another person verifies:

"Does John really need this access?"

---

# Real-world Example

A company has a server:

```
SERVER01
    |
    |
Domain Admin Account
```

The Domain Admin password is stored inside CyberArk.

A technician needs to fix a problem.

He does:

1. Opens PVWA
    
2. Finds Domain Admin account
    
3. Clicks "Connect"
    

CyberArk says:

```
Access requires approval
```

John enters:

```
Reason:
"Need Domain Admin access to fix DNS issue"
Duration:
2 hours
```

CyberArk sends approval request to an approver.

The approver sees:

```
User:
John

Account:
Domain Administrator

Reason:
Fix DNS issue

Duration:
2 hours
```

The approver clicks:

```
Approve
```

Now CyberArk allows John to use the account.

# Where is Dual Approval configured?

Usually in:

```
PVWA
 |
 Administration
 |
 Options
 |
 Access Control
 |
 Dual Control
```

or inside:

```
Safe Settings
```

depending on CyberArk version.

The idea is:

You define:

- Which accounts require approval
    
- Who can approve
    
- How many approvals are needed

# Dual Approval vs Dual Control

You may hear both terms.

They are related.

## Dual Control

The general security concept:

> Two people must participate in a sensitive operation.

Example:

```
Person A requests
Person B approves
```

---

## Dual Approval

A CyberArk implementation of dual control:

> Access to a privileged account requires approval before it is granted.

# Example with CyberArk Components

Let's connect it to CyberArk architecture:

```
User
 |
 |
 v
PVWA
(Web Interface)
 |
 |
 v
Vault
(Stores account secrets)
 |
 |
 v
Approval Engine
 |
 |
 v
Approver
 |
 |
 v
PSM
(Session Connection)
 |
 |
 v
Target Server
```

Important:

The approver **does not get the password**.

They only approve the request.

CyberArk still controls the password.

---

# What happens if approval is denied?

Example:

John requests:

```
Domain Admin
Duration: 8 hours
Reason: Testing
```

Manager thinks:

"John does not need this."

They click:

```
Reject
```

Result:

```
Request denied

John cannot access the account
```

The password never leaves CyberArk.

---

# What happens after approval expires?

Example:

Approval:

```
User:
John

Access:
Domain Admin

Time:
2 hours
```

After 2 hours:

```
Access automatically removed
```

John cannot continue using the account.

This is called:

**Just-In-Time (JIT) access**

# Dual Approval + PSM

Normally CyberArk wants you to use:

```
User
 |
 v
PSM
 |
 v
Server
```

not:

```
User
 |
 v
Password
 |
 v
Server
```

With Dual Approval:

```
Request
   |
Approval
   |
PSM session
   |
Recording
   |
Server
```

The session can still be:

- Recorded
    
- Monitored
    
- Audited

Think of a school computer lab:

Without Dual Approval:

```

"I want the teacher password"

System:
"Here you go"
```

Bad.

---

With Dual Approval:

```
Student:
"I need teacher password to install software"

```

Much safer.

# Summary

|Term|Meaning|
|---|---|
|Dual Approval|One person requests access, another approves|
|Purpose|Prevent misuse of powerful accounts|
|Requester|Person who needs privileged access|
|Approver|Person who verifies the request|
|Password|Never shown to approver|
|CyberArk components|PVWA + Vault + PSM|
|Benefit|Accountability and security|
|Common use|Domain Admin, Root, Database Admin accounts|

The main idea:

> **CyberArk Dual Approval adds a "second pair of eyes" before someone can use a powerful privileged account.**
