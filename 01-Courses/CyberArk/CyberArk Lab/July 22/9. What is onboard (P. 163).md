# What is **Onboard** in CyberArk?

Imagine your company has a **secret key** that opens a very important room. The key belongs to an administrator account like:

```
Username: Administrator
Password: P@ssw0rd123
Server: Windows Server 01
```

CyberArk does not protect this account until it knows about it.

**Onboarding** means:

> **Adding a privileged account into CyberArk so CyberArk can start protecting, managing, and controlling it.**

In simple words:

**Before onboarding:**

```
Administrator Account
        |
        |
        v
Outside CyberArk
(Password is not protected)
```

**After onboarding:**

```
Administrator Account
        |
        |
        v
CyberArk Vault
(Password stored securely)
        |
        |
        v
CPM manages password
PSM controls access
PVWA lets users request access
```

---

# Why do we need onboarding?

Companies have thousands of powerful accounts:

- Domain Administrator
    
- Windows Administrator
    
- Database Admin
    
- Root Linux accounts
    
- Service accounts
    
- Application accounts
    

These accounts are dangerous because they have high permissions.

Example:

A hacker steals:

```
Username:
Administrator

Password:
Summer2026!
```

The hacker can:

- Delete files
    
- Create new users
    
- Install malware
    
- Access sensitive data
    

CyberArk protects these accounts by putting them into the Vault.

# What happens during onboarding?

Let's go step-by-step.

## Step 1: Find the account

First, CyberArk needs to know:

"What account do you want me to protect?"

Example:

```
Account:
Administrator

Machine:
WindowsServer01

Domain:
Company.local
```

This account already exists in Active Directory.

---

## Step 2: Add the account to CyberArk

An administrator adds the account into PVWA.

PVWA:

```
Password Vault Web Access
```

It is the CyberArk website interface.

The account information is added:

```
Account Name:
Administrator

Address:
Server01.company.local

Username:
Administrator

Platform:
Windows Domain Account

Safe:
Windows Admin Safe
```

---

## Step 3: CyberArk stores the password

The password moves into the CyberArk Vault.

Before:

```
Windows Server
       |
       |
Password stored on server
```

After:

```
Windows Server
       |
       |
CyberArk Vault
       |
Encrypted password
```

The password is encrypted and protected.

---

## Step 4: CPM starts managing it

After onboarding, the **CPM** (Central Policy Manager) takes responsibility.

CPM asks:

"Should I change this password?"

Example policy:

```
Change password every:
30 days

Password length:
20 characters

Use:
Numbers
Symbols
Uppercase
Lowercase
```

Then CPM automatically changes:

```
Old Password:
Admin123

        ↓

New Password:
X8#kL92@pQ11!
```

The user does not need to know the password.

# What are the main parts involved in onboarding?

## 1. PVWA

**PVWA = Password Vault Web Access**

This is where administrators add accounts.

Example:

```
Admin
 |
 |
PVWA
 |
 |
Add Account
```

---

## 2. Vault

The Vault stores the secret.

Example:

```
Username:
Administrator

Password:
*************

Stored securely
```

---

## 3. CPM

**CPM = Central Policy Manager**

CPM manages passwords.

It:

- Changes passwords
    
- Verifies passwords
    
- Reconciles passwords if needed
    

---

## 4. PSM

**PSM = Privileged Session Manager**

PSM controls how users connect.

Example:

Without CyberArk:

```
User
 |
 |
Server
```

User knows the password.

---

With CyberArk:

```
User
 |
 |
PSM
 |
 |
Server
```

The user connects but never sees the password.

---

# Different ways to onboard accounts

## 1. Manual onboarding

An administrator adds accounts one by one.

Example:

```
Add Account
     |
Username: Administrator
Password: ********
```

Good for small environments.

---

## 2. Bulk onboarding

Many accounts are added at once.

Example:

```
1000 Windows accounts
        |
        |
CSV file
        |
        |
CyberArk
```

Good for large companies.

---

## 3. Discovery + onboarding

CyberArk can find accounts automatically.

Example:

CyberArk scans:

```
Active Directory
        |
        |
Find privileged accounts
        |
        |
Suggest onboarding
```

---

# What happens after onboarding?

After onboarding, users can:

### Request access

Example:

```
John needs Admin account

John
 |
Request Access
 |
CyberArk
 |
Approve
 |
Access granted
```

---

### Use password safely

CyberArk can:

- Show password temporarily
    
- Connect through PSM
    
- Record sessions
    

---

### Automatically rotate passwords

Example:

```
Monday:
Password = ABC123

30 days later:

Password = XZY987
```

No human needs to change it.

# In your CyberArk lab, when you see "Onboard Account", think:

**Onboard = Bring an account into CyberArk protection.**

The flow is:

```
Account exists
       |
       v
Discover account (optional)
       |
       v
Onboard account into PVWA
       |
       v
Store password in Vault
       |
       v
CPM manages password
       |
       v
PSM controls sessions
       |
       v
Account is protected
```

So the purpose of onboarding is:

✅ Put privileged accounts under CyberArk control  
✅ Secure their passwords  
✅ Automatically rotate passwords  
✅ Control who can use them  
✅ Create auditing and tracking of access
