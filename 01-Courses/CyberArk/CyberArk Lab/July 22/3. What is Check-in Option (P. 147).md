#cyberark-labs 


Imagine CyberArk is a **secure locker** that stores important passwords (accounts). A person can **borrow** a password from the locker for a limited time, use it, and then return it.

The **Check-in option** is basically the **"return the password now" button**.

---

## First: What happens when someone takes an account?

Imagine CyberArk has this account:

```
Server Admin Account
Username: Administrator
Password: ********
```

Tom needs to use it.

The normal process:

```
Tom
 |
 | 1. Requests access
 v
CyberArk PVWA
 |
 | 2. Gives Tom the password
 v
Tom uses the account
 |
 | 3. Account is now "checked out"
 v
CyberArk marks it as busy
```

The account is now **checked out**.

Think of a library:

```
Book status:
Available ❌
Borrowed by Tom ✅
```

CyberArk does the same with passwords.

---

# What does "Check-in" mean?

**Check-in = returning the account back to CyberArk.**

Example:

Tom requested the Administrator account for 1 hour.

CyberArk says:

```
Account:
Administrator

Status:
Checked out by Tom

Time remaining:
45 minutes
```

Tom finishes early.

Instead of waiting 45 minutes, Tom can click:

```
Check-in
```

CyberArk says:

```
Tom returned the account.

Account is available again.
```

---

# Why do we need Check-in?

Because sometimes a user finishes their work early.

Example:

Tom needs admin access:

```
9:00 AM - Requests account
9:05 AM - Finishes task
```

But the account was approved for:

```
9:00 AM - 10:00 AM
```

Without Check-in:

```
Account locked until 10:00 AM
```

With Check-in:

```
Account released at 9:05 AM
```

The next person can use it.

---

# What is "Minimum Validity Period"?

This is the automatic timer.

Example:

CyberArk policy:

```
Minimum Validity Period:
30 minutes
```

Meaning:

"When CyberArk gives a password to someone, keep it checked out for at least 30 minutes."

Example:

Tom gets the password:

```
10:00 AM
```

Minimum Validity:

```
30 minutes
```

At:

```
10:30 AM
```

CyberArk automatically releases it.

Because they want to demonstrate **automatic release**.

Instead of Tom clicking:

```

```

```
Minimum Validity Period expires
              |
              v

              |
              v
```

# Simple analogy

Think of a hotel room:

```
Room 101
```

Tom rents it.

The hotel says:

```
You have the room until 6 PM
```

Tom leaves at 3 PM.

Two choices:

### Option 1: Check-in manually

Tom tells the hotel:

> "I am done. You can give this room to someone else."

### Option 2: Automatic release

Tom does nothing.

At 6 PM:

> The reservation expires automatically.

CyberArk Check-in works the same way.

## In CyberArk words:

|Term|Meaning|
|---|---|
|Checkout|User takes control of an account/password|
|Checked out|Account is temporarily unavailable to others|
|Check-in|User returns the account manually|
|Unlock Account permission|Permission to force-release someone else's checkout|
|Minimum Validity Period|Time before CyberArk automatically releases the account|
|Automatic release|CyberArk returns the account without user action|

So in your lab, they are teaching you that **CyberArk can release accounts in two ways:**

1. **Manual Check-in** → user/admin clicks a button
    
2. **Automatic release** → CyberArk waits until the allowed time ends and releases it automatically ✅
