#cyberark-labs 


Imagine you work at a company and there is a **very powerful account** called:

```
Administrator
```

This account can:

- delete users
    
- change settings
    
- install software
    
- access important servers
    

The company does **not** want every employee to use this account whenever they want.

So they create a rule:

> "Before someone can use this powerful account, another person must approve it."

That is what an **Approval Request** is in CyberArk.

---

# What is an Approval Request?

An **Approval Request** is a request created by a user asking for permission to access a protected account or perform a privileged action.

Simple example:

```
User
 |
 |  "I need Administrator account access"
 ↓
CyberArk creates Approval Request
 |
 ↓
Manager/Security Admin reviews it
 |
 ↓
Approve or Reject
 |
 ↓
User gets access (if approved)
```

---

# Why do we need Approval Requests?

Because privileged accounts are dangerous.

Think about a house:

Your normal house key:

```
Bedroom key
```

Not very dangerous.

But a master key:

```
Master key
```

can open every door.

CyberArk protects these "master keys" (privileged accounts).

Before giving someone the master key, the company wants someone to say:

> "Yes, this person really needs it."

---

# Real-world example

A company has a server:

```
Production Database Server
```

Inside CyberArk:

```
Safe: Production Servers

Account:
    SQL_Admin
```

John is a developer.

John needs to fix a database problem.

Instead of giving John the password:

```
SQL_Admin password:
    P@ssw0rd123
```

CyberArk says:

"John, request access first."

John clicks:

```
Request Access
```

CyberArk creates:

```
Approval Request #1001

Requested by:
John

Account:
SQL_Admin

Reason:
Fix database issue

Duration:
2 hours
```

A database manager receives:

```
John wants SQL_Admin access.

Approve?
[ YES ] [ NO ]
```

---

# What happens after approval?

If the manager clicks:

```
APPROVE
```

CyberArk allows John to use the account.

Usually the process is:

```
John
 |
 | Request access
 ↓
PVWA
 |
 | Creates approval request
 ↓
Approver
 |
 | Approves
 ↓
PVWA
 |
 ↓
PSM
 |
 ↓
John connects to server
```

John might never see the password.

CyberArk can automatically:

- connect John through PSM
    
- provide a temporary session
    
- record the session
    
- hide the password

# Approval Request vs Normal Access

Let's compare.

## Without Approval

```
User
 |
 | Has permission
 ↓
Gets account password
 ↓
Uses account
```

Problem:

❌ Nobody checks why  
❌ No control  
❌ Password could be stolen

---

## With Approval

```
User
 |
 | Requests access
 ↓
Approval needed
 ↓
Manager approves
 ↓
Access granted
 ↓
CyberArk records everything
```

Benefits:

✅ Someone verifies the request  
✅ Company knows who used the account  
✅ Access can be temporary  
✅ Better security

---

# Who approves the request?

Usually:

- Manager
    
- Security team
    
- System owner
    
- Application owner
    

Example:

```
Developer requests:
    Linux root account

Approved by:
    Linux administrator
```

---

# Where do you see Approval Requests in CyberArk?

Usually in:

```
PVWA
 |
 └── Accounts
       |
       └── Request Access
```

or:

```
PVWA
 |
 └── My Requests
```

An administrator can configure approval workflows.

---

# How does CyberArk know approval is required?

Because of policies.

Example:

A Safe may have a rule:

```
Safe:
Production Servers

Permission:
Use Accounts

Require Approval:
YES

Approvers:
Security Team
```

Meaning:

Anyone trying to use accounts from this Safe must get approval.

---

# Approval Request vs Dual Control (important difference)

These two are related but different.

## Approval Request

One person requests access.

Example:

```
John:
"I need access"

Manager:
"Approved"
```

---

## Dual Control

Two people must be involved.

Example:

```
John:
"I need access"

Manager:
"Approved"

Security Admin:
"Also approved"
```

Two approvals are required.

# Simple analogy

Think of a bank vault:

### Normal account

You have your own wallet:

```
Your money
```

No approval needed.

---

### Privileged account

A bank vault:

```
Company secrets
Administrator passwords
Root accounts
```

Before opening it:

```
Person A:
"I need to open vault"

Person B:
"Why?"

Person B:
"Approved"
```

That is an **Approval Request**.

---

## In one sentence:

**A CyberArk Approval Request is a security checkpoint where a user asks permission to use a privileged account, and an authorized person approves or rejects the request before access is granted.**
