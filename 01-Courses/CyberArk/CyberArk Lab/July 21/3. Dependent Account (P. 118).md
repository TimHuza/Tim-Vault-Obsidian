#cyberark-labs 


Imagine CyberArk is a **super secure password manager for companies**.

It stores many accounts:

```
Administrator account
Database account
Service account
Application account
SSH account
```

Sometimes **one account depends on another account** to work. That is where **Dependent Accounts** come in.

---

## 1. What is a Dependent Account?

A **Dependent Account** is an account that needs another account's password to work correctly.

In simple words:

> A dependent account is an account that says: "I cannot do my job unless another account gives me its password."

The account that is needed is called the **Master Account** (or main account).

The dependent account "depends" on it.

---

## Simple Example (Real World)

Imagine a company has an application:

```
Banking Application
        |
        |
        needs password
        |
        v
Database Account
```

The application uses a database account to connect to the database.

Example:

```
Database Account:
Username: db_admin
Password: P@ss123
```

The application stores this password somewhere:

```
Application Config File:

database_user = db_admin
database_password = P@ss123
```

Now CyberArk manages the database password.

But what happens when CyberArk changes:

```
Old password:
P@ss123

New password:
Xy98!abc
```

CyberArk changes the password in the Vault.

But the application still has:

```
database_password = P@ss123
```

The application breaks.

Why?

Because the application does not know the password changed.

---

## 2. How Does Dependent Account Solve This?

CyberArk says:

"Before I change this password, I need to update the things that use this password."

So we create a dependency:

```
Database Account
        |
        |
        depends on
        |
        v
Application Account
```

Meaning:

```
Database Account = Parent

Application Account = Dependent Account
```

Now CyberArk knows:

"Before changing the database password, I must update the application."

---

## 3. CyberArk Password Change Process With Dependent Account

Without dependent account:

```
CyberArk CPM
     |
     |
Change password
     |
     v
Database password changed


Application
     |
     |
Uses old password ❌

Application breaks
```

---

With dependent account:

```
             CyberArk Vault
                  |
                  |
                  v

          Database Account
                  |
                  |
          Password change needed
                  |
                  |
                  v

        Check dependencies

                  |
                  |
                  v

        Update Application
        with new password

                  |
                  |
                  v

        Change database password

                  |
                  |
                  v

        Application works ✅
```

# 4. Why Do We Need Dependent Accounts?

Because many systems store passwords that are used by other systems.

Examples:

---

## Example 1: Windows Service Account

Imagine:

```
Windows Server

Service:
  Backup Service

Runs as:
  DOMAIN\backup_user
```

The service uses:

```
backup_user password
```

CyberArk changes:

```
backup_user password
```

But the Windows service still has the old password.

Result:

```
Backup Service stops ❌
```

Dependent account helps CyberArk update the service.

---

## Example 2: Database Application

```
Web Application

      |
      |
      uses

      v

SQL Database Account
```

The web app needs the database password.

CyberArk must update the application after password rotation.

---

## Example 3: Scheduled Tasks

Windows:

```
Scheduled Task

Runs as:

DOMAIN\report_user
```

CyberArk rotates:

```
report_user password
```

The scheduled task needs the new password.

# 5. What Does CyberArk Store?

A normal account:

```
Safe
 |
 |
 +-- Domain Admin Account

Username:
administrator

Password:
********
```

A dependent account:

```
Safe
 |
 |
 +-- Database Account
 |
 |
 +-- Dependent Account
        |
        |
        +-- Application
        +-- Service
        +-- Scheduled Task
```

It tells CyberArk:

"This account is connected to another account."

---

# 6. Dependent Account vs Linked Account

Many beginners confuse these.

## Dependent Account

Question:

> "Who uses this password?"

Example:

```
Database Account
       |
       |
       v
Application
```

The application depends on the database account password.

---

## Linked Account

Question:

> "Which account helps manage another account?"

Example:

```
Domain Admin Account
          |
          |
          v

Windows Local Admin Account
```

CyberArk uses one account to manage another.

---

Easy memory:

|Concept|Meaning|
|---|---|
|Dependent Account|Something that uses another account's password|
|Linked Account|An account connected to another account for management|

---

# 7. Where Does CPM Fit?

The **CPM (Central Policy Manager)** is responsible for changing passwords.

Normal rotation:

```
CPM
 |
 |
Change password
 |
 |
Account
```

With dependent accounts:

```
CPM

 |
 |
Checks dependencies

 |
 |
Updates dependent systems

 |
 |
Changes password

 |
 |
Everything continues working
```

# Simple 12-year-old explanation:

Imagine you have a house:

```
House key = password
```

You give the key to:

- your parents
    
- your brother
    
- your dog walker
    

If you change the lock:

```
Old key ❌
New key ✅
```

Everyone who uses the key needs the new key.

A **Dependent Account** is CyberArk's list saying:

> "These people also use this key. Update them when the key changes."

---

In CyberArk labs, you will usually see Dependent Accounts with:

- Windows services
    
- Scheduled tasks
    
- Applications
    
- Databases
    
- Scripts
    

because these are the things that break when passwords rotate.
