# 1. What is a Dependent Account?

A **Dependent Account** is an account that **stores a password/secret somewhere else** and depends on another account.

Example:

You have an application:

```
Web Application
       |
       |
       uses password
       |
       v
Database Account
```

The application has a configuration file:

```
app.config

DatabaseUser = app_user
DatabasePassword = Password123
```

The application needs this password to connect to the database.

So CyberArk manages:

```
CyberArk Account:
-----------------
Database Account
Username: app_user
Password: Password123
```

But the password is also stored inside:

```
Application configuration file
```

That application configuration is a **dependent account**.

---

# 2. Why do we need Dependent Accounts?

The problem:

CyberArk changes the password:

Before:

```
Database Account password:

Password123
```

CyberArk rotates it:

```
New password:

X9!fK22@abc
```

Now:

CyberArk knows:

```
Database Account
Password = X9!fK22@abc
```

But the application still has:

```
app.config

Password123
```

The application breaks.

Why?

Because the application is using an old password.

# 3. What does "Link Account" mean in Dependents?

When you go:

```
PVWA
 |
 |-- Account
       |
       |-- Dependents
              |
              |-- Add Dependent
                     |
                     |-- Link Account
```

You are telling CyberArk:

> "This dependent object belongs to this account and must be updated when this account changes."

Example:

Main account:

```
Database Account

Username:
db_service

Password:
Password123
```

Dependent:

```
Application Configuration File

Location:
C:\app\config.ini
```

Relationship:

```
                CyberArk

                    |
                    |
        -------------------------
        |                       |
        v                       v

 Database Account        Application Config

 Password123             Password123

        |
        |
        Password changes
        |
        v

 X9!fK22@abc

        |
        |
        v

 Update dependent automatically

```

# 4. Now the important part:

# What does "Sync Secret" button do?

The blue **Sync Secret** button means:

> **Copy the current password/secret from the main account to the dependent account.**

It forces CyberArk to synchronize the secrets immediately.

---

Imagine this situation:

Main account:

```
SQL_Service

CyberArk password:

NewPassword123
```

Dependent:

```
Application config file:

OldPassword456
```

They are different.

CyberArk sees:

```
Main Account:
NewPassword123

Dependent:
OldPassword456
```

They are out of sync.

---

When you click:

```
Sync Secret
```

CyberArk says:

"Take the secret from the main account and push it to the dependent."

Result:

Before:

```
Database Account
----------------
Password:
NewPassword123


Application Config
------------------
Password:
OldPassword456
```

After:

```
Database Account
----------------
Password:
NewPassword123


Application Config
------------------
Password:
NewPassword123
```

Now they match.

# 5. Real-life example

Imagine a Windows service.

You have:

## Main Account

```
DOMAIN\SQL_Service

Password:
ABC123
```

CyberArk manages it.

---

The service configuration contains:

```
Windows Service:

Log On As:
DOMAIN\SQL_Service

Password:
ABC123
```

This is the dependent.

---

CyberArk rotates:

```
ABC123
        |
        v
NEW987!
```

Now:

```
CyberArk Vault:
NEW987!

Windows Service:
ABC123
```

The service stops.

---

Click:

```
Sync Secret
```

CyberArk updates:

```
Windows Service:

Password:
NEW987!
```

Service starts again.

# 6. Sync Secret vs Automatic Rotation

Important difference:

## Sync Secret

You manually press:

```
Sync Secret
```

It means:

> "Synchronize now."

Used for testing or fixing problems.

---

## Automatic Synchronization

Normally CyberArk does this automatically:

```
CPM rotates password
          |
          |
          v
CPM updates dependents
          |
          |
          v
Applications continue working

```

You don't press the button every time.

---

# 7. Linked Account vs Dependent Account (very important)

Many beginners mix these.

## Linked Account

Question:

> "Which account should CyberArk use to manage this account?"

Example:

```
Domain Admin
       |
       |
       v
SQL_Service
```

Domain Admin helps change SQL_Service password.

---

## Dependent Account

Question:

> "Where else is this password used?"

Example:

```
SQL_Service
       |
       |
       v
Application Config File
```

The application depends on the password.

---

# Simple memory trick:

### Linked Account:

**"Who helps CyberArk manage this account?"**

```
Helper Account
      |
      v
Managed Account
```

---

### Dependent Account:

**"Who depends on this account's password?"**

```
Managed Account
      |
      v
Application / Service
```

---

### Sync Secret:

**"Make the dependent password equal to the main account password."**

```
Main Account Secret
          |
          |
          v
Dependent Secret
```

So in your lab, when you click **Dependents → Add Dependent → Link Account → Sync Secret**, you are testing the process of making sure the dependent system receives the same updated secret as the account stored in CyberArk.
