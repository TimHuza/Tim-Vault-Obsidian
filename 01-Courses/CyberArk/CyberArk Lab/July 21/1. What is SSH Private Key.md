# First, what is SSH?

Imagine you have a computer called **Database Server**.

```
Your PC  -------------------->  Database Server
```

But the database server doesn't let just anyone connect.

Before it lets you in, it asks:

> **"Who are you? Prove it!"**

This is called **authentication**.

There are two common ways to prove who you are:

1. Password
    
2. SSH Private Key
    

---

# What is an SSH Private Key?

Think of it like a **very special secret key**.

Imagine a treasure chest.

```
       Treasure Chest
      +--------------+
      |              |
      |              |
      +--------------+
            🔒
```

Only one special key can open it.

```
      🔑 SSH Private Key
```

The server has a lock.

You have the matching key.

If the key matches...

```
Server:
"Yep, that's the correct key."

✔ Access granted
```

If the key doesn't match...

```
❌ Access denied
```

---

# Why not just use a password?

Passwords can be:

```
Password:
Summer2026!
```

Someone might:

- guess it
    
- steal it
    
- accidentally see it
    

An SSH key looks nothing like a password.

Example:

```
-----BEGIN OPENSSH PRIVATE KEY-----

b3BlbnNzaC1rZXktdjEAAAAABG5vbmUAAAAE...

-----END OPENSSH PRIVATE KEY-----
```

It is a long encrypted piece of data.

Nobody can realistically guess it.

That's why SSH keys are **much more secure**.

---

# SSH actually has TWO keys

This is the part beginners usually don't know.

SSH uses **two keys**, not one.

```
           Pair of Keys

        🔑 Private Key
             +
        🔓 Public Key
```

They are created together.

Think of them as:

```
House Lock
+
House Key
```

---

## Public Key

The server keeps this.

```
Database Server

authorized_keys

Public Key
```

It is okay if people see it.

---

## Private Key

Only the user owns this.

```
CyberArk

Private Key
```

Nobody else should ever have it.

It must stay secret.

---

# How do they work?

Imagine this.

```
Database Server
-----------------------
Has Public Key
```

CyberArk has

```
Private Key
```

When CyberArk connects...

```
CyberArk
    |
    |  Uses Private Key
    |
    v
Database Server
```

The server checks:

```
Does this private key match my public key?
```

If yes...

```
✔ Login
```

If not...

```
❌ Reject
```

Notice something important:

**The private key is never sent to the server.**

Instead, SSH uses cryptography to prove that CyberArk has the correct private key without exposing it.

# Why is this useful for databases?

Many database administrators log in through SSH before working with the database.

Example:

```
CyberArk

↓

SSH

↓

Linux Server

↓

Oracle Database
```

Instead of remembering passwords, the administrator uses an SSH key.

CyberArk can store that key securely in the Vault.

---

# Why store the SSH Private Key inside CyberArk?

Imagine the private key is saved on someone's laptop.

```
Laptop

Private Key
```

If the laptop is stolen...

The attacker now has the key.

Not good.

Instead:

```
CyberArk Vault

🔒 SSH Private Key
```

Only CyberArk can retrieve it when needed.

Benefits:

- ✔ More secure
    
- ✔ Encrypted
    
- ✔ Audited
    
- ✔ Controlled access
    
- ✔ Easy to rotate or replace
    

---

# Why do we use an SSH Private Key for a database connection?

Because many databases (especially Oracle, PostgreSQL, MySQL, etc.) run on Linux servers.

CyberArk often needs to:

- connect to the Linux server
    
- change passwords
    
- verify passwords
    
- run maintenance commands
    
- start password rotation
    

Using an SSH private key allows CyberArk to authenticate securely without relying on a password.
