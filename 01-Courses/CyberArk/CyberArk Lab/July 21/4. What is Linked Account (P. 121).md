#cyberark-labs 


A **Linked Account** in CyberArk means:

> **One CyberArk account is connected (linked) to another CyberArk account because one account needs the other account to perform an action.**

Think of it like a **key and a helper key**.

---

## 1. The problem Linked Accounts solve

Imagine you have a Windows server.

There are two accounts:

### Account 1: Application Account

```
Username: SQL_Service
Purpose: Runs SQL Database service
```

This account is used by SQL Server.

---

### Account 2: Administrator Account

```
Username: CyberAdmin
Purpose: Has permission to change passwords
```

This account can reset the SQL_Service password.

---

Now CyberArk wants to automatically change:

```
SQL_Service password
```

But there is a problem:

CyberArk tries:

```
CyberArk CPM
        |
        |
        v
Change SQL_Service password
```

The server says:

> "Wait! Only an administrator can change this password."

So CyberArk needs another account:

```
CyberAdmin
```

to help it.

This is where **Linked Account** comes in.

---

# 2. What is a Linked Account?

A Linked Account tells CyberArk:

> "To manage this account, use another account."

Example:

```
Main Account:
-----------------
SQL_Service
(password needs rotation)


Linked Account:
-----------------
CyberAdmin
(can perform password change)
```

Relationship:

```
CyberAdmin
     |
     |
     | helps manage
     v
SQL_Service
```

---

# 3. Real-world analogy

Imagine you have a safe.

Inside the safe:

```
Box A:
Your bicycle key

Box B:
Master key
```

You want to change the lock on Box A.

But you cannot open Box A without the master key.

So you tell someone:

> "Use the master key from Box B to access Box A."

The master key is like the **Linked Account**.

---

# 4. How Linked Account works in CyberArk

Let's follow the process.

## Step 1: Account is stored in CyberArk

Example:

```
Safe: Production Servers

Account:
SQL_Service
```

CyberArk stores:

```
Username: SQL_Service
Password: ********
```

---

## Step 2: You configure a Linked Account

You tell CyberArk:

```
SQL_Service
     |
     |
     needs
     |
     v
Domain_Admin
```

Meaning:

"Use Domain_Admin when you need to manage SQL_Service."

---

## Step 3: CPM tries to change password

The CPM (Central Policy Manager) says:

```
I need to change SQL_Service password
```

It checks:

```
Does SQL_Service need another account?
```

CyberArk answers:

```
Yes.
Use Domain_Admin.
```

---

## Step 4: CyberArk uses the linked account

The flow becomes:

```
              CyberArk Vault
                    |
                    |
              gets passwords
                    |
                    v

              CPM Server

                    |
                    |
        Login using Linked Account

                    |
                    v

             Target Server

                    |
                    |
        Change SQL_Service password

```

---

# 5. Where do we configure Linked Accounts?

Usually:

```
PVWA
 |
 |-- Accounts
        |
        |-- Select Account
              |
              |-- Linked Accounts
```

You choose:

```
Main Account:
SQL_Service

Linked Account:
Domain_Admin
```

# 6. Common examples of Linked Accounts

## Example 1: Windows Service Accounts

Very common.

A service account:

```
SQL_Service
```

cannot change its own password.

Administrator:

```
Domain_Admin
```

does it.

Relationship:

```
Domain_Admin
      |
      |
      v
SQL_Service
```

---

## Example 2: Oracle Database Accounts

Database user:

```
Oracle_App
```

needs password rotation.

But only:

```
Oracle_Admin
```

can change passwords.

So:

```
Oracle_Admin
      |
      |
      v
Oracle_App
```

---

## Example 3: Application dependencies

Imagine:

```
Application Server
       |
       |
needs password
       |
       v

Database Account
```

If the database password changes, the application must update too.

A linked account can help CyberArk manage this dependency.

---

# 7. Linked Account vs Reconcile Account

Beginners often confuse these.

## Linked Account

Purpose:

> "Use another account during normal management."

Example:

```
Change password normally:

Admin Account
      |
      v
Service Account
```

---

## Reconcile Account

Purpose:

> "Fix a broken password."

Example:

CyberArk thinks:

```
Password = ABC123
```

but the server has:

```
Password = XYZ789
```

CyberArk cannot login.

A reconcile account fixes it:

```
Reconcile Account
        |
        v
Reset password
        |
        v
Service Account
```

# 9. Why do we need Linked Accounts?

Without Linked Accounts:

❌ CyberArk cannot change some passwords  
❌ Password rotation fails  
❌ Manual administrators are needed  
❌ Security risk increases

With Linked Accounts:

✅ Password rotation becomes automatic  
✅ No humans need to know admin passwords  
✅ Least privilege is maintained  
✅ Service accounts stay secure

---

## The easiest way to remember:

**Linked Account = "The account CyberArk borrows to manage another account."**

or:

> **The main account is the patient. The Linked Account is the doctor that helps treat the patient.**

In CyberArk labs, you will usually see Linked Accounts with **service accounts, domain accounts, database accounts, and accounts that cannot change their own passwords.**
