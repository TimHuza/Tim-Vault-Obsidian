# First, where are these settings?

These settings belong to a **Platform**.

Remember:

> A **Platform** is like a **rule book** that tells CyberArk how to manage a certain type of account.

Example:

- Windows Platform
    
- Linux Platform
    
- Oracle Platform
    

Every account using that platform follows these settings.

---

# 1. ImmediateInterval

You found it here:

```
Platform
└── UI & Workflows
    └── Automatic Password Management
        └── General
             ImmediateInterval = 1
```

---

## What is it?

This tells CyberArk:

> **"If someone presses 'Change Password Immediately', how long should CyberArk wait before actually doing it?"**

Normally CyberArk changes passwords according to a schedule.

For example:

```
Every 30 days
```

But sometimes an administrator says:

> "No! Change it NOW!"

CyberArk puts that request into the CPM queue.

The **ImmediateInterval** tells CPM how often to check for these urgent requests.

---

## Imagine a pizza shop 🍕

Imagine you ordered a pizza.

Normally:

```
Pizza orders are checked every 30 minutes.
```

But then someone says:

> "This order is URGENT!"

The workers look at the urgent order every **1 minute**.

That's exactly what ImmediateInterval does.

---

## Example

Administrator clicks:

```
Change Password Immediately
```

CyberArk does **not** instantly start.

Instead:

```
CPM
↓

Checks every 1 minute

↓

"Oh! There is an immediate request."

↓

Starts changing password.
```

If it were:

```
ImmediateInterval = 10
```

CPM could wait up to **10 minutes** before noticing.

---

## Why do we need it?

Because checking every second would waste resources.

CyberArk simply asks:

> "Any urgent jobs?"

every X minutes.

---

## In your screenshot

```
ImmediateInterval = 1
```

means

> CPM checks every **1 minute** for immediate password change requests.

---

# 2. AllowedSafes

You found:

```
AllowedSafes = -LIN-S-
```

---

## What is a Safe?

A Safe is like a secure vault.

Example:

```
Windows Safe
Linux Safe
Oracle Safe
Network Safe
```

Each Safe stores passwords.

---

## What does AllowedSafes mean?

It tells CyberArk:

> "This Platform may only be used inside these Safes."

---

Imagine a school 🏫

Think of Platforms as different teachers.

```
Linux Teacher

Windows Teacher

Oracle Teacher
```

And classrooms are Safes.

```
Room A = Linux Safe

Room B = Windows Safe

Room C = Oracle Safe
```

You don't want the Linux teacher teaching in the Windows classroom.

So you say:

```
Linux Teacher

Allowed Classroom:

Linux Room only
```

Exactly the same idea.

---

## Example

Suppose you have:

```
Platform:
Linux Platform
```

AllowedSafes:

```
-LIN-S-
```

Then accounts can only exist in

```
-LIN-S-
```

If someone tries putting a Linux account into

```
WindowsSafe
```

CyberArk can prevent it.

---

## Why do we need it?

Without it:

```
Linux Platform

↓

Could accidentally be used everywhere.
```

That becomes messy.

AllowedSafes helps organize everything.

# 3. SessionRecorderSafe

Second screenshot:

```
Privileged Session Management

SessionRecorderSafe
```

## First, what is PSM?

Remember:

```
User

↓

PSM

↓

Target Server
```

The user never connects directly.

Everything goes through the PSM server.

---

While the user works,

PSM records everything.

Think of it like a security camera.

```
Login

Mouse

Keyboard

Commands

Windows opened

Everything
```

---

## Where are recordings stored?

CyberArk needs somewhere safe.

That's exactly what **SessionRecorderSafe** is.

---

Imagine a security camera 📹

A security camera records video.

But where does the video go?

It goes into a hard drive.

```
Camera

↓

Hard Drive
```

CyberArk works the same.

```
PSM

↓

Recording

↓

SessionRecorderSafe
```

---

## Example

Administrator connects to:

```
Linux Server
```

through

```
PSM
```

PSM records:

```
Login

Commands

Terminal

Logout
```

When finished:

```
Recording

↓

Stored inside

↓

PSMR_LinSsh30
```

That Safe now contains the recording.

---

## Why not store recordings on the PSM server?

Because:

- recordings could fill up the disk
    
- someone could delete them
    
- attackers might steal them
    
- backups become difficult
    

Instead:

```
PSM

↓

Uploads recording

↓

Vault Safe
```

The Vault protects it.

---

## Why do we need SessionRecorderSafe?

Without it:

```
PSM records session

↓

???

↓

Nowhere to save it
```

CyberArk must know exactly which Safe stores recordings.

---

```

```

```

```

# Beginner Summary

|Parameter|Simple Meaning|Why We Need It|
|---|---|---|
|**ImmediateInterval**|How often CPM checks for urgent password change requests|So urgent password changes happen quickly without constantly checking every second|
|**AllowedSafes**|Which Safes are allowed to use this Platform|Keeps the correct account types in the correct Safes and prevents mistakes|
|**SessionRecorderSafe**|The Safe where PSM stores session recordings|Keeps recorded sessions secure, organized, and protected inside the Vault|

As you continue learning CyberArk, you'll notice a pattern:

- **CPM** manages **passwords**.
    
- **PVWA** is the **web interface** where administrators configure policies.
    
- **PSM** manages and **records sessions**.
    
- **The Vault (Safes)** securely stores **passwords, recordings, and other sensitive data**.
    
- **Platforms** define **how** CyberArk should manage a particular type of account.
