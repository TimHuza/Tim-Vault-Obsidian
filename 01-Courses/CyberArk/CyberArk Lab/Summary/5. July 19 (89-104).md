#cyberark-labs 


Continuing with your CyberArk journey, here is a page-by-page summary of pages 89 to 104, where you move from watching security tapes to solving the problem of "locked doors" using linked accounts.

### **Page 89: Closing the Session and Starting the Audit**

- **The Summary:** You finish your work as Carlos by typing `exit` to close your secure connection.
- **The Switch:** A new section starts called **Auditor Tasks**. You stop being the worker (Carlos) and log in as **Cindy**, the company's security guard, to check what people have been doing.

### **Page 90: Finding the "Security Footage"**

- **The Summary:** Still acting as Cindy, you go to the **Monitoring** tab.
- **The Task:** You look through a list of all the digital "recordings" of past sessions. You find the recording for the session Carlos just finished.

### **Page 91: Reviewing the Tapes**

- **The Summary:** You click **Play** to watch a video of exactly what happened on Carlos's screen.
- **The Details:** You see that the system doesn't just record video; it also keeps a list of every command Carlos typed, like a transcript, so you can jump to specific moments in the video.

### **Page 92: The Oracle Mission Begins**

- **The Summary:** A new chapter starts: **Securing Oracle Database Accounts**.
- **The Architect Task:** You log in as **Mike** (the administrator) to create a new "instruction manual" (Platform) specifically for Oracle database keys named **Ora Dba 30**.

### **Page 93 & 94: Fine-Tuning the Database Rules**

- **The Summary:** On these pages, you finish setting up the Oracle rules.
- **The Warning:** The guide reminds you never to copy and paste directly from the PDF into the lab, as it can cause weird errors.

### **Page 95: Building the Oracle Vault**

- **The Summary:** You switch to **Paul** (the Safe Manager) to create a new digital room (Safe) specifically for database keys.

### **Page 96: Adding the Oracle Database Key**

- **The Summary:** You put the database account (`dba01`) into the Safe.
- **What’s Different:** Unlike a regular computer account, for a database, you have to tell CyberArk extra details like the **Port number** and the name of the **Database** it belongs to.

### **Page 97: Troubleshooting the Robot**

- **The Summary:** As we discussed in our last chat, this is where you run into a timeout error because the robot (CPM) needs more time to talk to the tricky Oracle server. You fix it by increasing the **Timeout** setting to 90 seconds.

### **Page 98: Introduction to "Linked Accounts"**

- **The Summary:** You start a new chapter on **Linked Accounts**.
- **The Problem:** Sometimes, a computer is so secure it won't let a powerful "root" user log in directly from the outside—the door is locked from the inside.

### **Page 99: Making the "Key for the First Door"**

- **The Summary:** To get around the "locked door" problem, you start creating a **Logon Account** platform. This is like a smaller, regular key you use to get in the front door before you can use the bigger, "root" key.

### **Page 100: Sorting the Video Tapes**

- **The Summary:** You learn about the **`SessionRecorderSafe`** setting.
- **The Goal:** You tell the system to save all video recordings for these specific accounts into a folder named after the platform (like `PSMR_LinLogon30`) so the Auditor can find them easily later.

### **Page 101: Building the Safe for Logon Keys**

- **The Summary:** You create a new Safe named **`P-BOS-LIN-S-LOGON`**.
- **The Strategy:** You keep these "regular" logon keys in a separate room from the "powerful" root keys for better security.

### **Page 102: Setting Up Permissions**

- **The Summary:** You add the **Linux Admins** group to the new Safe.
- **The Rule:** You give them **"Connect only"** rights, meaning they can use the keys to open doors, but they can't see or change the secrets themselves.

### **Page 103: Attaching the Keys Together**

- **The Summary:** You find the powerful `root01` account and click the **Details** tab.
- **The Magic Step:** You "Link" the regular `logon01` account to the powerful `root01` account. Now CyberArk knows that to use the big key, it must use the small key first.

### **Page 104: The "Jump Account" Success**

- **The Summary:** You log in as Carlos and test the connection.
- **The Result:** When you click Connect, you see a PuTTY window pop up where the system automatically logs in as the small account (`logon01`) and then immediately switches (jumps) to the powerful account (`root01`) for you!
